# gnFortress 엔타이틀먼트(실행 한도) 설계서
### 셀프서비스 하이브리드 모델 — 티어별 실행 범위 정의

> 용도: 셀프서비스 전환에 따른 "티어별로 고객이 직접 실행할 수 있는 범위"의 제품 스펙.
> 연계: 가격 전략은 `pricing_packaging.md`, 화면은 랜딩(A), 가입 흐름은 온보딩(C).
> ⚠️ 수치는 모두 **가설(예시)** — 원가·경쟁·전환데이터로 검증 후 확정.

---

## 1. 기본 원칙

1. **DIY(셀프) + 한도(엔타이틀먼트)**: 고객이 자기 화면에서 직접 점검·리포트 생성. 티어가 "할 수 있는 범위"를 결정.
2. **하이브리드 모션**: Starter·Pro는 셀프서비스 즉시 가입, **Enterprise는 영업·계약**(무제한·관제·SLA).
3. **부드러운 한도(graceful)**: 한도 초과 시 기능을 깨지 말고 "업그레이드 유도"로 전환.
4. **소유권 우선**: 어떤 티어든 고객이 **자기 AWS 계정에 읽기전용 역할을 배포**해야만 점검 가능(C 참조).

## 4. 한도 초과 시 동작 (Enforcement) — 차원별

| 차원 | 한도 도달 시 UX |
|---|---|
| D1 계정 추가 | "계정 한도(1/1) 도달 — **Pro로 업그레이드 시 10개까지**" 모달 + 업그레이드 버튼. 기존 계정은 정상 동작 |
| D2 HostIPS 배포 | Starter: 메뉴 자체 비활성 + "Pro에서 제공". Pro(5/5): "한도 초과 — Enterprise 상담" |
| D3 스케줄 | 상위 주기 옵션은 자물쇠 아이콘 + 툴팁 "Pro 이상" |
| D4 온디맨드 | 월 한도 소진 시 "이번 달 잔여 0회 — 다음달 초기화 또는 업그레이드". **하드 차단(남용 방지)** |
| D6 뮤트 | Starter: 버튼에 🔒 + 업셀 |
| D8 seat 초대 | 초대 시 "seat 한도 도달" |
| 공통 원칙 | **기존 데이터·기능은 절대 깨지 않음**. 추가 행위만 차단 + 업그레이드 동선 노출 |

## 5. 업그레이드 / 다운그레이드 흐름

**업그레이드 (Starter → Pro)**: 셀프서비스 즉시. 결제 등록 → 한도 즉시 상향(계정 1→10 등). 일할 정산.

**업그레이드 (Pro → Enterprise)**: **셀프 불가 → "상담 신청"**. 무제한·관제·SLA·온프렘은 계약·보안검토 필요.

**다운그레이드 (Pro → Starter)**: 초과분 선(先)정리 필요.
- 예: 계정 7개 사용 중 → Starter(1개)로 내리려면 **6개를 먼저 해제**해야 적용. "초과 리소스를 정리해야 다운그레이드됩니다" 가이드.
- 데이터 보관도 30일로 축소됨을 사전 경고.

**해지**: 읽기전용 역할 제거 안내 + 데이터 파기 정책 고지(보관기간 후 삭제).

## 6. Free / Trial 정책 (택1 권고)

| 옵션 | 내용 | 장점 | 단점 |
|---|---|---|---|
| **A. Freemium** (권고) | Starter 1계정 **영구 무료**(기능 제한) | 마찰 0, 리드 풀 극대화, 자가 확산 | 무료 사용자 비용 부담 |
| B. Free Trial | Pro 기능 **14일 무료** 후 유료/Starter 강등 | 상위 기능 체험 → 전환↑ | 신용카드 선등록 시 마찰 |
| C. 둘 다 | Starter 영구무료 + "Pro 14일 체험" 버튼 | 최상 전환 깔때기 | 운영 복잡 |

→ **권고: A(Freemium)로 시작**, 데이터 쌓이면 C로 확장. "무료 진단" 리드 자석을 **"무료로 직접 1계정 점검"**으로 진화.

## 7. 기술 구현 메모 (제품·엔지니어링)

- **plan → entitlements 매핑 테이블**(서버 진실 소스): `{plan: {d1_accounts, d2_hostips, d3_schedule, ...}}`. 코드에 하드코딩 금지, **DB/설정으로 관리**(영업이 Enterprise 한도 커스텀 가능하게).
- **enforcement 위치**: 백엔드 API에서 강제(프론트 가드는 UX용일 뿐). 계정 추가·HostIPS 배포·스케줄 생성·온디맨드 실행 시 한도 체크.
- **사용량 집계**: 현재 사용량(계정 N/한도 M) 실시간 표시 → 대시보드 상단 "사용량 게이지".
- **소유권 검증 게이트**(C와 연계): 점검 실행 전 "역할 배포 확인" 필수.
- **남용 방지**: D4 온디맨드 rate limit + 신규 가입 시 첫 스캔 전 업무 이메일 인증(개인메일 차단 옵션).

## 8. 한도 수치 결정 논리 (확정 전)
- D1(계정): Starter 1은 "맛보기", Pro 10은 "중견기업 다계정 평균", Enterprise 무제한. → 실제 타깃 고객 계정 분포 조사로 보정.
- D2(HostIPS): Pro 5는 "핵심 워크로드만". 초과 수요가 Enterprise 전환 트리거가 되도록 의도적으로 낮게.
- D4(온디맨드): 남용·비용 방지용. 정상 사용엔 충분, 자동화 남용엔 부족하게.
