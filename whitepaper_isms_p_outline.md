## 1. 목차 (Table of Contents)

### ▎표지 + Executive Summary (1~1.5쪽)
- **목적**: 바쁜 의사결정자(CISO)가 첫 장만 읽어도 핵심을 파악하게.
- **핵심 내용**:
  - "클라우드 ISMS-P는 일회성 점검이 아니라 *상시 운영*의 문제다"라는 한 줄 주제.
  - 백서가 답하는 3가지 질문 미리보기: ① 왜 어려운가 ② 무엇을 점검해야 하는가 ③ 어떻게 상시 대비하는가.
- **삽입 자료**: 핵심 메시지 3-포인트 박스, 통계 1개(예: 클라우드 도입률·인증 의무대상 증가 추이).

### ▎1장. ISMS-P 인증, 왜 지금 더 중요한가 (2쪽)
- **목적**: 독자의 "위기감" 자극 = 읽어야 할 이유 부여.
- **핵심 내용**:
  - ISMS-P 인증이란 무엇인가 (정보보호+개인정보보호 통합 관리체계 인증, KISA 주관).
  - **의무대상**: 정보통신망법상 의무대상 기준(매출·이용자 수 등) 요약.
  - 미인증/결함 시 리스크: 과태료·과징금, **B2B 계약 입찰 자격 박탈**, 사고 시 평판.
  - 클라우드 전환이 이 문제를 키운 이유 (온프렘 점검 방식이 안 통함).
- **삽입 자료**: 의무대상 판별 플로차트, 리스크 요약 표.

### ▎2장. 클라우드(AWS)에서 ISMS-P가 어려운 5가지 이유 (2쪽)
- **목적**: "그래서 기존 방식으론 안 된다"는 공감대 형성.
- **핵심 내용** (5가지):
  1. **책임 공유 모델(Shared Responsibility)** — AWS가 지키는 영역 vs 고객이 지켜야 할 영역 혼동.
  2. **동적 환경** — 리소스가 매일 생성·삭제·변경 → 분기 1회 수동 점검으론 시점이 안 맞음.
  3. **멀티 계정 복잡성** — 50~500 계정 규모에서 일관된 점검·집계 난이도.
  4. **항목 매핑의 어려움** — "S3 버킷 공개 차단"이 ISMS-P 어느 통제에 해당하는지 사람이 매번 판단.
  5. **증빙의 부담** — 심사 때 "고쳤다"를 증명할 이력·근거 수집이 수작업.
- **삽입 자료**: AWS 책임 공유 모델 도식, "수동 점검의 시점 공백" 타임라인 그림.

### ▎3장. ISMS-P 최신 인증체계 바로 이해하기 (2~2.5쪽)
- **목적**: 독자에게 실질 지식 제공(= 전문성 증명).
- **핵심 내용**:
  - 인증기준 3대 영역 구조: ① 관리체계 수립·운영 ② 보호대책 요구사항 ③ 개인정보 처리단계별 요구사항.
  - 이 중 **클라우드 기술 점검과 직접 맞닿는 통제 영역**이 무엇인지 강조 (접근통제·암호화·로그/감사·취약점 관리 등).
  - ISMS vs ISMS-P 차이 한 줄 정리.
- **삽입 자료**: 인증기준 영역 구조도, "클라우드 자동점검으로 커버 가능한 항목 vs 정책·문서로 증빙할 항목" 구분 표. ※개수는 KISA 고시 대조 후 확정.

### ▎4장. AWS 설정 ↔ ISMS-P 통제 매핑 (핵심 체크리스트) (3쪽 — *백서의 하이라이트*)
- **목적**: 실무자가 **저장·인쇄해서 쓰고 싶은** 가장 실용적인 장. 다운로드 동기의 핵심.
- **핵심 내용**:
  - 대표 AWS 점검 항목을 ISMS-P 통제에 매핑한 **실전 체크리스트** (15~25개 발췌).
  - 예시 행: `S3 퍼블릭 액세스 차단` → 접근통제 / `RDS 저장암호화` → 암호화 / `CloudTrail 활성화` → 로그·감사 / `보안그룹 0.0.0.0/0 차단` → 네트워크 접근통제.
  - 각 항목: 점검 의미 · 심각도(높음/보통/낮음) · 위반 시 인증 영향.
- **삽입 자료**: **체크리스트 표**(점검항목 | ISMS-P 통제 | 심각도 | 확인 방법). 가디언넷 186개 항목 중 대표 발췌임을 명시.

### ▎5장. 수동 점검 vs 자동화 점검, 무엇이 다른가 (1.5쪽)
- **목적**: 자동화(=제품 카테고리 CSPM)로의 인식 전환.
- **핵심 내용**:
  - 비교 표: 점검 주기 / 누락 위험 / 인력 비용 / 시점 정확도 / 증빙 / 멀티계정 확장성.
  - "사람이 분기 1회"의 구조적 한계 → "매일 자동 + 상시 모니터링".
- **삽입 자료**: 좌우 비교 표, 점검 커버리지·시점 정확도 비교 그래프.

### ▎6장. 발견에서 "조치완료"까지 — 발견사항 생명주기 관리 (2쪽 — *차별화 핵심*)
- **목적**: 단순 스캔을 넘는 "추적"의 가치 = 가디언넷만의 강점을 일반론으로 풀어 설명.
- **핵심 내용**:
  - "취약점을 찾는 것"과 "고쳤는지·다시 터졌는지 추적하는 것"은 다르다.
  - 생명주기 4상태 개념: **미조치 → 조치완료 → 재발 → (뮤트)**.
  - 왜 심사에 중요한가: "한 번 고쳤다 또 열린 항목(재발)"을 못 잡으면 **감사 증빙이 무너진다**.
  - 1취약점 = 1행으로 추적해 점검을 반복해도 노이즈가 안 쌓이는 원리.
- **삽입 자료**: 생명주기 상태 전이 다이어그램(미조치→조치완료→재발).

### ▎7장. 오탐·예외 관리(뮤트)로 노이즈 줄이기 (1쪽)
- **목적**: 실무자의 현실 고충(알람 피로) 해결 = 공감.
- **핵심 내용**:
  - 모든 취약 항목이 "지금 당장 고칠 것"은 아니다 — 의도된 예외·오탐 존재.
  - **뮤트(인지된 예외)** 개념: 삭제가 아니라 "음소거" → 진짜 위험에 집중, 점수·KPI 왜곡 방지.
  - 뮤트도 기록이 남아 심사 시 "왜 예외 처리했는가"를 설명 가능.
- **삽입 자료**: 뮤트 처리 전후 대시보드 노이즈 비교 이미지.

### ▎8장. "상시 인증 태세"를 만드는 운영 모델 (1.5쪽)
- **목적**: 벼락치기 → 상시 운영으로의 전환 제시.
- **핵심 내용**:
  - 주기적(스케줄) 자동 점검 → 컴플라이언스 점수 추세 관리 → 경영진 정기 보고.
  - "감사 전날 야근"이 사라지는 구조.
  - 멀티 계정·서비스별 현황을 한 화면에서 관리.
- **삽입 자료**: 컴플라이언스 점수 추세 그래프(예시), 운영 사이클 다이어그램(점검→추적→보고→재점검).

### ▎9장. ISMS-P 인증 준비 단계별 로드맵 (2쪽)
- **목적**: 독자에게 **실행 계획**을 손에 쥐여줌 = 백서 최고의 효용.
- **핵심 내용**: 6개월 타임라인 예시
  - 1개월차: 범위 정의·자산 식별 / 2개월차: 현황 진단(베이스라인 점수) /
    3~4개월차: 결함 조치·재점검 / 5개월차: 증빙 정리 / 6개월차: 심사 대응.
  - 각 단계에서 자동화 점검이 절약하는 시간·인력.
- **삽입 자료**: **간트 차트형 로드맵**(다운로드·인쇄 유도용 핵심 비주얼).

### ▎10장. gnFortress로 구현하기 (1.5쪽 — *유일하게 제품을 전면화*)
- **목적**: 앞선 9장의 "이상"을 실제로 구현하는 방법 = 자연스러운 제품 연결.
- **핵심 내용**:
  - 가디언넷 소개: 국내 금융권 1위, 4,000+ 워크로드 보호 실적.
  - gnFortress 삼중 방패: **Posture(점검·ISMS-P) / Workload(실시간 방어) / 관제(XDR)**.
  - 본 백서의 6단계 운영 모델을 gnFortress가 어떻게 그대로 구현하는지 매핑.
  - 186개 ISMS-P 자동 점검 · 생명주기 추적 · 뮤트 · 스케줄 · 컴플라이언스 점수.
- **삽입 자료**: 제품 화면 캡처(점수 대시보드·발견사항·스케줄), 6단계↔기능 매핑 표.

### ▎11장. 부록 (Appendix) (2쪽)
- **A. ISMS-P 클라우드 자가진단 체크리스트** (독자가 바로 써보는 15문항 표 — *재방문·공유 유발*).
- **B. 용어집** — 양호/취약/해당없음, 미조치/조치완료/재발/뮤트, CSPM/CWPP/CNAPP, 심각도 3단계.
- **C. 참고 자료** — KISA ISMS-P 안내, AWS 보안 모범사례 링크.
- **D. 무료 진단 안내 + 문의처** (랜딩페이지 QR/URL, 연락처).
